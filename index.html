01 — TCP CONNECTION

TCP 연결과 소켓

▍한 줄 요약

DB 연결은 클라이언트와 서버 사이에 맺어진 하나의 TCP 소켓이며, 데이터를 주고받기 전에 먼저 3-way handshake로 신뢰성 있는 양방향 통로를 세운다.

애플리케이션이 데이터베이스에 "연결한다"는 것은, 운영체제 수준에서 보면 (클라이언트 IP, 클라이언트 포트, 서버 IP, 서버 포트) 네 값으로 식별되는 TCP 소켓 하나를 여는 일이다. 서버는 정해진 포트에서 듣고 있다(PostgreSQL 5432, MySQL 3306). TCP는 신뢰성·순서보장·흐름제어를 제공하는 연결 지향 프로토콜이므로, 실제 데이터 한 바이트를 보내기 전에 양쪽이 시퀀스 번호를 교환하며 통로를 합의하는 3-way handshake(SYN → SYN-ACK → ACK)를 거친다. 이 핸드셰이크가 끝나야 비로소 그 위에서 DB 고유의 대화가 시작된다.

근거 / WHY
  • 왜 TCP인가 — DB 쿼리·결과는 손실·재정렬되면 안 되는 데이터다. TCP는 재전송·순서보장으로 이를 보장하므로, 거의 모든 DB가 기본 전송 계층으로 TCP를 쓴다(같은 호스트면 Unix domain socket을 쓰기도 한다).
  • 핸드셰이크의 의미 — SYN/SYN-ACK/ACK는 양쪽의 초기 시퀀스 번호와 윈도우 크기를 합의하는 과정이다. 즉 최소 1 RTT(왕복 지연)를 "데이터 전송 이전"에 무조건 지불한다.
  • 연결은 상태다 — 한번 맺힌 소켓은 명시적으로 닫기(FIN/ACK 4-way 또는 RST) 전까지 양쪽 OS의 연결 테이블에 남아 자원을 점유한다. 이 "상태성"이 뒤의 풀링·누수·죽은 연결 문제의 근원이다.
02 — THE REAL COST

연결 수립의 진짜 비용

▍한 줄 요약

새 DB 연결 하나의 비용은 TCP 핸드셰이크가 전부가 아니라 TLS + 인증 + 세션 초기화로 이어지는 여러 번의 네트워크 왕복, 그리고 연결마다 서버가 프로세스/스레드를 띄우는 유지 비용까지 포함한다.

"연결이 비싸다"는 말의 실체를 분해해 보면 두 종류의 비용이 있다. 첫째는 수립 비용: TCP 1 RTT 위에 TLS 핸드셰이크(TLS 1.2면 +2 RTT, 1.3이면 +1 RTT)가 얹히고, 그 다음 DB 프로토콜 핸드셰이크 — PostgreSQL의 StartupMessage → SCRAM-SHA-256 인증 왕복 → AuthenticationOk/ParameterStatus/ReadyForQuery, MySQL은 서버가 먼저 greeting을 보내는 식 — 이 또 여러 왕복을 더한다. 결과적으로 첫 쿼리를 보내기도 전에 4~6번의 왕복이 일어날 수 있다. 둘째는 유지 비용: PostgreSQL은 연결마다 OS 프로세스를 fork하고(process-per-connection), MySQL은 스레드를 할당한다. 연결이 놀고 있어도 메모리와 슬롯을 먹으며, max_connections를 넘기면 FATAL: too many clients로 거절된다.

클라이언트 서버 왕복(RTT) 구간
근거 / WHY
  • RTT는 곱해진다 — 각 왕복은 네트워크 지연에 비례한다. 같은 리전 수 ms라도 핸드셰이크가 4~6회면 수십 ms. 초당 수천 요청에서 매 요청마다 새 연결을 맺으면 이 지연이 그대로 처리량 천장이 된다.
  • 유지 비용이 더 치명적일 때가 많다 — PostgreSQL의 process-per-connection 모델에서 수백 개의 연결은 수 GB 메모리와 컨텍스트 스위칭으로 이어진다. 그래서 "연결 수" 자체가 DB의 확장 한계가 된다.
  • 서버리스가 이 문제를 극대화한다 — Lambda처럼 인보케이션마다 연결을 새로 여는 구조는 동시 실행 수만큼 연결을 폭증시켜 순식간에 max_connections를 고갈시킨다. (→ 07장 외부 풀러)
03 — CONNECTION POOL

Connection Pool란

▍한 줄 요약

커넥션 풀은 미리 맺어둔 연결들을 보관해 두고 빌려주고(borrow) 돌려받으며(return) 재사용하는 캐시로, 비싼 연결 수립 비용을 요청마다 치르지 않게 해준다.

연결 수립이 비싸다면 답은 단순하다 — 매번 새로 만들지 말고, 만들어 둔 것을 재사용하자. 커넥션 풀은 일정 개수의 살아있는 연결을 메모리에 보관하는 자료구조다. 애플리케이션은 풀에서 연결을 빌려(acquire) 쿼리를 실행하고, 끝나면 풀에 반납(return)한다. 여기서 가장 중요한 개념 전환: 풀 환경에서 connection.close()는 TCP를 실제로 끊는 것이 아니라 "풀에 돌려준다"는 의미다. 연결의 비싼 부분(핸드셰이크·인증·프로세스 fork)은 풀이 처음 채워질 때 한 번만 치르고, 이후 수천 건의 요청이 그 비용을 공유한다. 대표 구현으로 Java의 HikariCP, Python의 SQLAlchemy QueuePool, Go의 database/sql 내장 풀, Django의 CONN_MAX_AGE 등이 있다.

매번 핸드셰이크 재사용(저렴)
근거 / WHY
  • 비용의 분할 상환(amortization) — 핸드셰이크라는 고정 비용을 풀의 수명 동안 발생하는 모든 요청이 나눠 갖는다. 요청이 많을수록 요청당 연결 비용은 0에 수렴한다.
  • 연결 수 상한 통제 — 풀은 동시에 열리는 연결 수에 천장(maximumPoolSize)을 둔다. 트래픽이 폭증해도 DB가 받는 연결 수가 폭증하지 않아, max_connections 고갈을 막는다.
  • 예측 가능성 — 연결 수립 지연(꼬리 지연)이 요청 경로에서 사라지므로 p99 레이턴시가 안정된다.
04 — POOL MECHANICS

풀의 동작 메커니즘

▍한 줄 요약

풀은 "유휴 연결이 있으면 빌려주고, 없으면 상한까지 새로 만들고, 상한에 닿으면 타임아웃까지 대기시키며, 반납 시엔 세션 상태를 리셋해 되돌리는" 규칙으로 동작한다.

풀에 연결을 빌려달라는 요청(acquire)이 들어오면 세 갈래로 갈린다. ① 유휴(idle) 연결이 있으면 그것을 건넨다(선택적으로 빌려주기 전 유효성 검사). ② 유휴가 없지만 maximumPoolSize 미만이면 새 연결을 만들어 건넨다. ③ 이미 상한에 도달했으면 connectionTimeout만큼 대기하며, 그 안에 누가 반납하지 않으면 예외를 던진다(대기 큐). 사용이 끝나 반납될 때 풀은 그냥 꽂아 넣지 않고 세션 상태를 리셋한다 — 열린 트랜잭션을 롤백하고, 세션 변수를 초기화해 "깨끗한 연결"을 다음 사용자에게 보장한다.

idle (대기 중) in-use (대여 중) 대기 큐

핵심 파라미터

파라미터 (HikariCP 기준)의미흔한 기본/권장

근거 / WHY
  • 반납 시 리셋이 필수인 이유 — 이전 사용자가 SET한 세션 변수, 미완료 트랜잭션, 임시 테이블이 남으면 다음 사용자가 오염된 상태를 물려받는다. 그래서 풀은 반납 시 깨끗한 상태로 되돌린다.
  • 대기 큐의 존재 이유 — 상한을 두면 트래픽 폭증 시 초과 요청은 DB를 더 누르는 대신 앱 단에서 짧게 기다린다. "느린 DB"보다 "빠른 대기"가 전체적으로 낫다(→ 06장).
  • 고정 크기 권장 — HikariCP는 minimumIdle를 maximumPoolSize와 같게 둬 풀을 고정 크기로 운영하길 권한다. 동적 축소·재생성의 churn과 지연 스파이크를 없애기 위함이다.
05 — CONNECTION HEALTH

연결 관리와 건강성

▍한 줄 요약

오래 보관된 풀 연결은 방화벽·LB·DB 타임아웃에 의해 조용히 죽을 수 있으므로, maxLifetime·keepalive·유효성 검사로 선제 폐기하고, 누수 탐지로 반납되지 않는 연결을 잡아낸다.

풀이 연결을 재사용한다는 것은 곧 연결을 오래 살려 둔다는 뜻이고, 여기서 새로운 문제가 생긴다. 풀에 가만히 있던 연결을 중간 경로의 방화벽·NAT·클라우드 로드밸런서(흔히 350초 유휴 한계)나 DB 자체의 유휴 타임아웃(MySQL wait_timeout 등)이 조용히 끊어버린다. 풀은 이 사실을 모른 채 멀쩡한 줄 알고 빌려주고, 빌려간 쪽은 끊긴 소켓으로 쿼리를 날리다 실패한다 — 이른바 죽은 연결(stale connection) 문제다. 이를 막기 위해 ① maxLifetime으로 연결을 일정 나이에 선제 폐기하고(인프라가 강제하는 한계보다 몇 초 짧게 설정하는 것이 정석), ② keepaliveTime으로 유휴 연결에 주기적 핑을 보내 생존을 확인·유지하며, ③ 빌려주기 전 Connection.isValid()로 검사한다. 한편 빌려가서 영영 돌려주지 않는 연결 누수(leak)는 leakDetectionThreshold로 탐지한다.

운영 정석: maxLifetime < (DB·LB·방화벽이 강제하는 가장 짧은 연결 수명) − 여유(수 초). 그래야 풀이 항상 인프라보다 먼저 연결을 정리해, 인프라가 끊은 죽은 연결을 빌려주는 사고를 원천 차단한다.
근거 / WHY
  • 네트워크는 연결의 죽음을 알려주지 않는다 — 중간 장비가 연결 상태를 GC하거나 RST 없이 떨어뜨리면, 양 끝단은 다음 패킷을 보내기 전까지 죽음을 모른다. 그래서 "보내보기 전 확인" 또는 "주기적 핑"이 필요하다.
  • 선제 폐기 > 사후 복구 — 죽은 연결을 잡아 재시도하는 것보다, 의심스러운 연결을 미리 버리고 새로 만드는 편이 꼬리 지연과 산발적 오류를 줄인다.
  • 누수는 천천히 풀을 고갈시킨다 — try-with-resources 누락 등으로 반납되지 않은 연결이 쌓이면 결국 풀이 비어 모든 요청이 connectionTimeout에 걸린다. 누수 탐지 로그가 코드 결함을 조기에 드러낸다.
06 — SIZING & OPTIMIZATION

풀 사이징과 최적화

▍한 줄 요약

동시 요청이 많다고 풀을 키우면 컨텍스트 스위칭·락 경합으로 오히려 처리량이 떨어지므로, DB 코어 수 기반 공식으로 상한을 잡고 Little's Law로 실수요를 추정해 "작은 풀 + 대기 큐"를 지향한다.

가장 반직관적인 대목이다. HikariCP 위키의 유명한 벤치마크는 연결 96개가 1000개(또는 4000개)보다 약 50배 빠를 수 있음을 보여준다. 이유는 단순하다 — DB의 실제 동시 작업 능력은 CPU 코어와 디스크에 묶여 있는데, 코어 수를 한참 넘는 연결을 던지면 그 연결들은 일을 하는 게 아니라 CPU를 두고 컨텍스트 스위칭, 락·래치 경합, 공유 버퍼 경합으로 서로를 방해한다. 그래서 처리량(TPS)은 작은 연결 수에서 정점을 찍고 그 이후엔 떨어진다. 출발점 공식은 다음과 같다.

# HikariCP 위키의 풀 사이징 공식 (출발점, 절대값 아님)
pool_size = (core_count × 2) + effective_spindle_count

# SSD라면 effective_spindle_count ≈ 1
#  4-core  →  (4 × 2) + 1 =  9
#  8-core  →  (8 × 2) + 1 = 17
# 16-core  → (16 × 2) + 1 = 33

# ⚠ core_count는 "DB 서버"의 코어다 — 앱 서버/쿠버네티스 노드가 아니다

Little's Law로 실수요 추정

공식이 상한을 준다면, Little's Law는 실제로 필요한 동시 연결 수를 추정해 준다. L = λ × W (평균 동시 처리 수 = 도착률 × 서비스 시간).

# 예: 초당 500 요청, 쿼리 평균 지연 10ms
L = λ × W = 500 req/s × 0.010 s = 5

# 평균적으로 동시에 5개 연결이면 수요를 감당한다는 추정.
# 여기에 변동 대비 여유를 약간 더해 풀 크기를 정하고,
# CPU 공식이 준 상한을 절대 넘기지 않는다.
근거 / WHY
  • 병렬성은 코어 수로 묶여 있다 — N개 코어는 동시에 N개 작업만 실제로 진행한다. 연결이 그보다 많으면 나머지는 OS 스케줄러에서 대기하며 스위칭 비용만 늘린다.
  • 대기 위치를 바꾸는 것이 핵심 — 어차피 기다려야 한다면, 비싸고 경합이 큰 DB 내부가 아니라 가볍고 빠른 앱 단의 풀 큐에서 기다리게 한다. 그래서 "작은 풀 + 큐"가 "큰 풀"을 이긴다.
  • 여러 서비스가 한 DB를 공유할 때 — 각 서비스 풀의 합이 DB의 max_connections 예산 안에 들어와야 한다. 서비스별로 풀을 무작정 키우면 합산이 DB를 무너뜨린다(→ 07장).
07 — EXTERNAL POOLER & MULTIPLEXING

외부 풀러와 다중화

▍한 줄 요약

인앱 풀이 한 앱의 연결 churn을 줄인다면, pgbouncer 같은 외부 풀러는 클라이언트-프록시-서버 구조로 수천 개의 클라이언트 연결을 소수의 서버 연결로 다중화해 DB 전체의 연결 수를 보호한다.

인앱 풀(HikariCP 등)은 앱 인스턴스 하나의 관점이다. 하지만 마이크로서비스 N개가 각자 풀 10개를 들면 DB는 수백 연결을 받고, 서버리스(Lambda)는 동시 실행 수만큼 연결을 폭증시킨다. 이때 pgbouncer 같은 외부 풀러가 앱과 DB 사이에 프록시로 들어가, 클라이언트 쪽으로는 많은 연결을 받아주면서 서버 쪽으로는 적은 수의 실제 연결만 유지해 둘을 다중화(multiplexing)한다. 즉 "클라이언트 연결의 수명"과 "서버 연결의 수명"을 분리한다. 다중화 정도는 풀링 모드로 결정된다.

클라이언트 연결 (많음) pgbouncer 프록시 서버 연결 (적음)

pgbouncer 풀링 모드

모드서버 연결을 점유하는 기간다중화제약

transaction 모드 주의: 연속된 두 트랜잭션이 같은 서버 연결로 간다는 보장이 없으므로, 세션에 묶이는 기능 — 일반 PREPARE/prepared statement, LISTEN/NOTIFY, 세션 단위 advisory lock, WITH HOLD 커서, 전역 SET — 이 깨진다. SET LOCAL처럼 트랜잭션 범위로 쓰거나, 애플리케이션이 모드에 협조해야 한다.
근거 / WHY
  • 계층이 다르다 — 인앱 풀(HikariCP)과 외부 풀러(pgbouncer)는 경쟁 관계가 아니라 다른 계층이다. 실무에선 앱 풀 → pgbouncer → DB로 함께 쓰며, 외부 풀러가 DB의 전체 연결 예산을 지킨다.
  • transaction 모드가 보통 정답인 이유 — 웹앱은 트랜잭션 사이에 비-DB 작업으로 연결을 놀린다. 트랜잭션 단위 반납은 이 유휴 시간을 회수해 적은 서버 연결로 많은 클라이언트를 감당하게 한다.
  • 서버리스의 구원자 — Lambda처럼 연결 churn이 극심한 환경에서는 외부 풀러(또는 RDS Proxy)가 사실상 필수다. 인보케이션이 폭증해도 DB가 받는 실제 연결 수를 일정하게 묶어준다.
★ — SELF CHECK

모범답안

OS 수준에서는 (클라이언트 IP·포트, 서버 IP·포트) 4-튜플로 식별되는 TCP 소켓을 하나 여는 것이다. 단계는 ① TCP 3-way handshake(SYN → SYN-ACK → ACK)로 신뢰성 통로 수립(1 RTT), ② (TLS 사용 시) TLS 핸드셰이크(+1~2 RTT), ③ DB 프로토콜 핸드셰이크 — 예: PostgreSQL StartupMessage → 인증 챌린지(SCRAM-SHA-256) 왕복 → AuthenticationOk/ParameterStatus/ReadyForQuery. 이 전부가 끝난 뒤에야 첫 쿼리를 보낼 수 있어, 총 4~6회의 왕복이 발생할 수 있다.

Q2"DB 연결 비용은 곧 TCP 핸드셰이크 비용이다"라는 주장의 어디가 불완전한가?모범답안 보기
모범답안

두 가지가 빠졌다. 첫째, 수립 비용은 TCP 1 RTT만이 아니라 TLS와 DB 인증·세션 초기화까지 합한 여러 왕복이다. 둘째이자 더 중요한 점은 유지 비용이다. PostgreSQL은 연결마다 OS 프로세스를 fork하고(MySQL은 스레드) 메모리·슬롯을 점유하므로, 연결이 놀고 있어도 자원을 먹고 max_connections 한계에 부딪힌다. 즉 비용은 "한 번의 핸드셰이크"가 아니라 "수립 + 살아있는 동안 내내의 유지"다.

Q3풀을 사용하는 코드에서 connection.close()를 호출하면 실제로 무슨 일이 일어나는가?모범답안 보기

TCP 연결을 실제로 끊는 것이 아니라 연결을 풀에 반납한다. 풀은 반납받은 연결의 세션 상태를 리셋(열린 트랜잭션 롤백, 세션 변수 초기화)한 뒤 유휴(idle) 상태로 되돌려 다음 요청이 재사용하게 한다. 핸드셰이크·인증 같은 비싼 비용은 다시 치르지 않는다.

Q6"동시 요청이 1,000개니까 풀도 1,000개로 키우자"가 왜 틀린 전략인지, 무슨 일이 벌어지는지 설명하라.모범답안 보기
모범답안

DB의 실제 병렬 처리 능력은 CPU 코어·디스크에 묶여 있다. 코어 수를 한참 넘는 연결은 일을 하기보다 CPU를 두고 컨텍스트 스위칭, 락·래치 경합, 공유 버퍼 경합으로 서로를 방해한다. 그래서 처리량(TPS)은 작은 연결 수에서 정점을 찍고 그 이후엔 떨어진다(HikariCP 벤치마크: 96개가 1,000~4,000개보다 ~50배 빠른 사례). 올바른 전략은 작은 풀로 상한을 두고, 초과 요청은 가벼운 앱 단 큐에서 짧게 기다리게 하는 것이다.

Q7DB 서버가 8코어(SSD)이고, 측정 결과 초당 800요청·쿼리 평균 5ms다. (a) 공식 상한과 (b) Little's Law 수요를 각각 계산하고 풀 크기를 제안하라.모범답안 보기
모범답안

(a) 공식 상한: (8 × 2) + 1 = 17. (b) Little's Law: L = 800 × 0.005 = 4 → 평균 약 4개 연결이면 수요를 감당. 따라서 실수요(≈4)에 변동 대비 여유를 약간 더하되, 공식 상한 17을 넘기지 않는 선 — 예컨대 8~12 정도가 합리적 출발점이다. 부하 테스트로 acquire 대기 시간과 TPS를 보며 미세 조정한다. (이 코어 수는 앱 서버가 아니라 DB 서버 기준임에 유의.)

Q8pgbouncer의 transaction 모드와 session 모드의 핵심 차이는? 고동시성 웹앱에 transaction 모드를 쓸 때 주의할 점은?모범답안 보기
모범답안

session 모드는 서버 연결을 클라이언트 세션 전체 동안 점유해 가장 안전하지만 다중화가 적다. transaction 모드는 트랜잭션 동안만 점유하고 commit/rollback 시 반납하므로, 트랜잭션 사이 유휴 시간을 회수해 적은 서버 연결로 많은 클라이언트를 감당한다(웹앱 표준). 단, 연속 트랜잭션이 같은 서버 연결로 간다는 보장이 없어 세션 단위 기능 — 일반 prepared statement/PREPARE, LISTEN/NOTIFY, 세션 advisory lock, WITH HOLD 커서, 전역 SET — 이 깨진다. SET LOCAL 사용, 프로토콜 수준 prepared statement 지원 설정, 짧은 트랜잭션 설계 등으로 모드에 협조해야 한다.
